# Building a Git-Backed Notebook on a Static Site

A record of what was built, the decisions behind it, and the traps to watch for.

---

## Key Features Implemented

### Authentication
- GitHub Personal Access Token entered via an in-page input on first visit
- Token saved to `localStorage` under `gh_pat` — never written to source code
- Token is verified against the GitHub Contents API before being stored (rejects invalid/scope-less tokens)
- A dismissable re-enter dialog (🔑 button) lets you rotate the token without wiping your current state

### Notebook & Note Management
- **Notebooks** = folders under `docs/` in the repository; creating one writes a `_category_.json` so Docusaurus auto-picks it up in the sidebar
- **Notes** = `.md` files inside those folders; files starting with `_` are excluded from the UI
- Create, open, edit, and delete notes entirely through the GitHub Contents API — no server needed
- Slugified filenames derived from the note title on first save; subsequent saves preserve the original filename

### Editor
- Title field + Markdown textarea
- **Ctrl+S / Cmd+S** triggers save
- **Render Markdown toggle** (react-markdown): switches between raw textarea and a rendered preview; in preview mode the title input is hidden and the title is shown as a plain heading above the rendered content
- **Close Note** button returns to the note list without saving

### Save Flow (modal-driven)
- Save opens a centered modal overlay with a progress bar
- **Phase 1 — Push**: PUT to GitHub Contents API; progress bar animates to ~45%
- **Phase 2 — Sync**: polls `GET /contents/docs/{notebook}` with a cache-busting `_=timestamp` query param until the saved file appears; bar fills to 100%
- On success the modal auto-dismisses after 1 second; the note stays open
- On network/API error the modal stays open with the error message and a Close button

### Merge Conflict Handling (409)
- If GitHub returns 409 (SHA mismatch), the modal closes and an amber banner appears inside the editor
- The latest SHA is auto-fetched from GitHub and loaded into state so a retry will succeed
- Banner has **Retry Save** (re-triggers save with the new SHA) and **Dismiss** buttons

### Delete Flow
- Trash icon (🗑) appears on each note row on hover
- Clicking shows an inline "Delete? Yes / No" confirmation on that row — no popups
- On confirm, calls `DELETE /contents/docs/{notebook}/{file}` with the blob SHA
- Same fill-bar + polling pattern as save: bar shows at top of note list and fills until the file is gone from the API response

### Sync Fill Bar
- A 3px progress bar sits between the note list header and the note list body
- Shared polling logic (`startSyncPoll`) accepts a `mode` parameter: `'appeared'` (post-save) or `'gone'` (post-delete)
- Polls every 2 seconds, up to 8 attempts; fills to ~85% while waiting, snaps to 100% on condition met, then fades out after 600ms

## Key Decisions

| Decision | Rationale |
|---|---|
| Static site + GitHub Contents API only | No server, no serverless functions, no secrets in source. Token lives only in the user's browser. |
| `localStorage` for PAT | Secure enough for single-user personal site; token never touches the network except to call GitHub's own API. |
| `react-markdown` over `marked` / `markdown-it` | No `dangerouslySetInnerHTML`, no DOMPurify dependency, native React elements, fits Docusaurus's existing remark/unified ecosystem. |
| Wrap `react-markdown` in `<BrowserOnly>` | Docusaurus pre-renders pages server-side; `react-markdown` v9+ is ESM-only and uses browser APIs. `BrowserOnly` prevents the SSR crash cleanly. |
| Poll with `?ref=main&_=timestamp` cache buster | GitHub's CDN can cache Contents API responses; without the cache buster the poll would keep seeing stale data and never resolve. |
| Keep note open after save | Avoids disruptive context switch; the modal gives clear save confirmation without navigating away. |
| SHA from directory listing for delete | The `GET /contents/docs/{notebook}` directory listing includes each file's `sha` blob field — no extra round-trip needed for deletes. |
| Auto-fetch latest SHA on 409 | A full three-way merge is complex and rarely needed for personal notes. Fetching the latest SHA and re-presenting the editor for a manual review is the lightest correct fix. |
| Slugify filenames, preserve on update | Prevents filenames from changing every time you rename a note's title after first save. |
| `_category_.json` for new notebooks | Docusaurus's `autogenerated` sidebar only picks up folders that contain at least one file; the category JSON also sets the sidebar label cleanly. |

## Gotchas

### GitHub Contents API quirks
- **SHA is required for updates**: every PUT to an existing file must include the current blob `sha` or GitHub returns 409. Always re-read the SHA after a successful save (`response.content.sha`) and store it in state for the next edit.
- **Directory listing vs file fetch**: a GET on a folder returns an array; a GET on a file returns an object with a `content` field (base64). The list items include `sha` but not `content`.
- **Base64 line breaks**: GitHub wraps base64 content at 60 characters with `\n`. Strip those newlines before passing to `atob`, otherwise decoding fails.
- **Cache busting on poll**: without `?_=Date.now()` or similar, the browser (and GitHub's CDN) may return a cached 200 showing the old state, causing infinite polling.

### Docusaurus + React specifics
- **ESM-only packages**: Docusaurus SSR will crash on `import` of a browser-only or ESM-only package (like `react-markdown` v9+). Use `<BrowserOnly>` or dynamic `require()` inside the `BrowserOnly` render prop.
- **`autogenerated` sidebar**: new docs folders only appear in the Docusaurus sidebar after a rebuild and deployment. The notebook editor UI reflects GitHub's file tree immediately, but the public docs sidebar lags until CI deploys.
- **`onBrokenLinks: 'throw'`**: the default Docusaurus config throws a build error for broken internal links. If you rename or delete a doc that is cross-linked, the build will fail.

### UX / state management
- **`key` prop on `NoteEditor`**: passing `key={editingNote?.path ?? 'new'}` forces React to fully remount the editor when switching notes. Without it, `useState` inside the editor retains the previous note's title and content.
- **Stale closure in polling**: the poll callback captures `selectedNotebook` and `authHeaders` at the time it was created. Always pass `notebook` as an explicit parameter to the poll starter rather than reading it from outer scope, or the poll will query the wrong folder if the user switches notebooks mid-poll.
- **Ctrl+S default behavior**: browsers intercept Ctrl+S to open their native Save dialog. `e.preventDefault()` in the keydown handler is mandatory.
- **`setSaving` timing**: `setSaving(false)` must happen after the async poll resolves, not at the end of the synchronous `saveNote` function body — otherwise the Save button re-enables before the operation is complete.
